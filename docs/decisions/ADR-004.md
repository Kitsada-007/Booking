# ADR-004: Express 5 on Vercel Serverless

## Status
Accepted

## Date
2026-06-28

## Context
The backend API serves as the data layer for a Next.js frontend, deployed on Vercel. Key requirements:
- REST API with 15+ resource modules
- Must run in Vercel's serverless Node.js runtime
- Automatic error handling for async route handlers
- CORS support for frontend on a different domain
- Compatible with Express middleware ecosystem (cors, helmet, morgan)

## Decision
Use Express 5.2.1 on Vercel serverless, deployed via `@vercel/node` with the monorepo config in root `vercel.json`.

Architecture:
- Single Express app exported as default, imported by Vercel as a serverless function
- All API routes mounted under `/api/` prefix
- `app.listen()` guarded by `process.env.NODE_ENV !== 'production'` (Vercel handles the listener)
- `vercel.json` routes all paths to the Express handler

## Alternatives Considered

### Express 4
- Pros: Mature ecosystem, more documentation, more community middleware verified
- Cons: Does not catch async errors in route handlers — every handler must wrap in try/catch or use a wrapper library like `express-async-errors`; Express 5 is the current major version as of 2026
- Rejected: Express 5 catches async rejections automatically, eliminating boilerplate try/catch wrappers

### Hono (lightweight alternative)
- Pros: Faster cold starts, built for serverless, smaller bundle
- Cons: Smaller middleware ecosystem; team familiarity with Express; existing codebase already uses Express patterns
- Rejected: Migration cost outweighs cold-start benefits at current scale

### Next.js API routes (in frontend project)
- Pros: Single deploy, shared types, simpler infrastructure
- Cons: Vercel 10s function timeout blocks long-running reports; backend would be coupled to Next.js release cycle; cannot test API independently
- Rejected: Already decided in ADR-001 to decouple frontend and backend

### tRPC
- Pros: End-to-end type safety, no REST contract needed
- Cons: Tightly couples frontend and backend; complex to set up with serverless; not suitable for public API consumption
- Rejected: REST API is simpler and more flexible for a multi-client system

## Consequences
- Express 5 auto-forwards async rejections to error handler — try/catch in route handlers is unnecessary but harmless (our codebase uses both patterns)
- No custom error handler middleware is configured — the default Express 5 error handler sends stack traces in development, HTML error pages in production
- CORS middleware allows only the configured `FRONTEND_URL` origin
- Serverless cold starts include Prisma client initialization (~200-400ms)
- The `@vercel/node` builder compiles TypeScript on Vercel using its own build step
