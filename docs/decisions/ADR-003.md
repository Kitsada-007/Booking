# ADR-003: JWT Dual-Token Auth with OTP Password Reset

## Status
Accepted

## Date
2026-06-28

## Context
The system serves multiple user roles (admin, room_staff, boat_staff, member) and requires:
- Session management that works across browser refreshes and tab closures
- Role-based authorization on API endpoints
- Password reset flow that doesn't require email service configuration for development
- Google OAuth as a convenience login method
- Stateless auth (avoid server-side sessions for Vercel serverless)

## Decision
Use a JWT dual-token strategy:
- **Access token**: Short-lived (15 min), sent with every request via `Authorization: Bearer` header
- **Refresh token**: Longer-lived (7 days), stored in `localStorage`, used to silently obtain new access tokens
- **Password reset**: 6-digit OTP stored in database with 15-minute expiry, verified before allowing password change
- **Google OAuth**: Server-side token verification using `google-auth-library`
- **Roles**: Stored directly on the User model, checked via `requireRole('admin')` middleware

## Alternatives Considered

### Server-side sessions (express-session + Redis/Postgres)
- Pros: Session revocation, no token management on client
- Cons: Stateful — incompatible with Vercel serverless (multiple instances); requires Redis or DB-backed session store; adds latency per request
- Rejected: Serverless deployment makes stateless JWT the natural choice

### Single long-lived JWT (no refresh token)
- Pros: Simpler client logic
- Cons: Cannot revoke tokens without a blocklist; if stolen, the token is valid until expiry; shorter expiry = worse UX (frequent re-login)
- Rejected: Refresh token pattern balances security and UX

### Magic link password reset
- Pros: No OTP to type
- Cons: Requires working email service in development; email delivery is unreliable in Thai market (some providers block transactional emails)
- Rejected: OTP works with email stubbed to console.log during development

### OTP sent via SMS
- Pros: Works without email
- Cons: Requires SMS gateway integration; additional cost per message; phone number collection adds privacy concerns
- Rejected: Over-engineering for a system where staff manages most password resets

## Consequences
- Access tokens cannot be revoked server-side — compromise window is 15 minutes max
- Refresh tokens stored in localStorage are vulnerable to XSS — mitigated by CSP headers and no third-party scripts
- OTP valid for 15 minutes, stored in database withhashed value (bcrypt)
- Email sending is stubbed to `console.log` in development — production must configure a real email provider
- Google OAuth is optional — the system works with email/password authentication only
- Role-based middleware (`requireRole`) is applied per-route, with 400/403 responses on mismatch
