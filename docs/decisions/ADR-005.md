# ADR-005: Dual-Path Payment (Gateway Mock + Bank Transfer)

## Status
Accepted

## Date
2026-06-28

## Context
The resort serves Thai customers who use different payment preferences. The system must support:
- Online payment via payment gateway (credit card, PromptPay)
- Bank transfer with manual staff verification
- Booking flow that works without a real payment gateway during development
- Slip upload for bank transfer verification
- Thai payment gateway market (Omise, 2C2P are common)

## Decision
Implement a dual-path payment system:
- **Gateway path**: Creates a payment record with method `'gateway'` and returns a mock redirect URL (`https://mock-gateway.example.com/pay/{id}`). In production, this URL would point to Omise or 2C2P.
- **Bank transfer path**: Creates a payment record with method `'bank_transfer'`, returns a list of active bank accounts from the database, and later supports slip image upload for staff verification.
- **Status flow**: `pending → paid (gateway) or verified (bank_transfer)` — staff must manually verify bank transfer payments.

No real payment gateway integration is implemented — the gateway URL is a mock placeholder.

## Alternatives Considered

### Single gateway only (Omise)
- Pros: Simplified code, automatic payment confirmation
- Cons: Excludes customers who prefer bank transfer (common in Thailand); requires gateway account setup before development
- Rejected: Bank transfer is a hard requirement for the Thai market

### Single bank transfer only
- Pros: Simplest implementation
- Cons: Poor UX — customers must wait for manual verification; no instant booking confirmation; increases staff workload
- Rejected: Gateway path provides instant confirmation and better UX

### Stripe (not common in Thailand)
- Pros: Excellent developer experience, well-documented
- Cons: Limited Thai bank support; higher fees; many Thai customers don't have international credit cards
- Rejected: Not well-suited for the Thai market

### Full Omise/2C2P integration during development
- Pros: Production-ready from day one
- Cons: Requires merchant account setup; test credentials may have limitations; adds complexity during early development
- Rejected: Mock gateway + configuration flag allows development without real credentials

## Consequences
- Gateway payments are not actually processed — the mock URL is a placeholder
- Bank transfer payments require staff dashboard to check and manually confirm
- Slip upload UI is not yet implemented — staff verifies outside the system currently
- `Payment` model includes `method`, `status`, `slipImageUrl`, and `verifiedAt` fields for the full flow
- Adding a real gateway requires implementing a webhook handler that calls the payment completion API
- The bank account management UI (CRUD for active accounts) is already built in the settings page
