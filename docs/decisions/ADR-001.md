# ADR-001: Decoupled Monorepo Structure

## Status
Accepted

## Date
2026-06-28

## Context
The project is a full-stack resort management system with a Next.js frontend and an Express API backend. Both need to be developed, tested, and deployed independently.

Key requirements:
- Frontend and backend have different dependency lifecycles (Next.js vs Express)
- Frontend and backend deploy to separate Vercel projects (different domains)
- Backend needs TypeScript compilation (tsc); frontend uses Next.js built-in compiler
- Shared types between frontend and backend must be duplicated (no shared package)
- Developer should be able to run frontend and backend independently during development

## Decision
Use a monorepo-style structure with completely decoupled frontend/backend directories, each with their own `package.json`, `tsconfig.json`, and build pipeline. No shared packages or workspace configuration.

```
vide-code/
├── frontend/          # Next.js 16 + React 19 + Tailwind 4
│   ├── package.json
│   └── src/
├── backend/           # Express 5 + Prisma 7 + Zod 4
│   ├── package.json
│   └── src/
├── docs/              # Specs, plans, decisions
└── AGENTS.md
```

Frontend calls backend via HTTP at runtime using `NEXT_PUBLIC_API_URL`. No compile-time coupling.

## Alternatives Considered

### Single Next.js app with API routes inside `/pages/api` or `/app/api`
- Pros: Single deploy, shared types, simpler setup
- Cons: Tight coupling — API logic cannot be tested or run independently of Next.js; Vercel serverless limits (10s timeout) on API endpoints unsuitable for complex report generation; Prisma client would run in same process as Next.js, causing cold start bloat
- Rejected: API routes hit Vercel's 10s timeout ceiling for reports; decoupling allows the backend to be scaled independently

### Turborepo / Nx monorepo with shared packages
- Pros: Shared types, coordinated builds
- Cons: Over-engineering for a team of one; adds toolchain complexity; Vercel deployment requires each workspace to be a separate project anyway
- Rejected: Not worth the overhead for the current team size

### Backend as a separate service in a different repo
- Pros: Cleanest separation
- Cons: More complex CI/CD; harder to track related changes across repos
- Rejected: Single repo simplifies cross-referencing changes and coordination

## Consequences
- Types must be duplicated between frontend and backend (e.g., API response interfaces)
- No TypeScript project references across packages
- Frontend and backend can be versioned, deployed, and scaled independently
- A shared `AGENTS.md` at the root documents conventions for both
- API contract (`docs/spec/api-contracts.md`) serves as the source of truth for data shapes
- Each sub-project has its own build and test commands
