# ADR-002: PostgreSQL + Prisma 7 ORM with Driver Adapter

## Status
Accepted

## Date
2026-06-28

## Context
The resort management system requires a relational database with strong integrity guarantees. Key requirements:
- Complex relationships (users, bookings, payments, reviews, rooms, boats, schedules)
- Transactional integrity for booking + payment creation
- Enum support for booking status lifecycle (pending_payment → confirmed → checked_in → completed)
- Type-safe queries — minimize runtime errors from malformed queries
- Schema migrations that can be reviewed and rolled back
- Serverless deployment (Vercel) requires connection pooling via external pooler (PgBouncer)

## Decision
Use PostgreSQL with Prisma 7 ORM, using the `@prisma/adapter-pg` driver adapter and the `pg` Node.js driver.

Connection setup:
```typescript
const pool = new Pool({ connectionString: process.env.DATABASE_URL });
const adapter = new PrismaPg(pool);
export const prisma = new PrismaClient({ adapter });
```

## Alternatives Considered

### MongoDB with Mongoose
- Pros: Schema-flexible, easy to start, well-known
- Cons: Our data is inherently relational (bookings reference users, rooms, payments); would need manual joins or denormalization; no native enum support for status transitions; no migration tooling as mature as Prisma Migrate
- Rejected: Relational data model makes a document store the wrong choice

### SQLite (via Turso/libsql)
- Pros: Zero-config, fast reads, edge-ready
- Cons: Limited concurrency; single-writer model unsuitable for booking system; no managed PostgreSQL-compatible hosting on Vercel marketplace
- Rejected: Not suitable for a multi-user production booking system

### Raw SQL with `pg` driver (no ORM)
- Pros: Full control, no ORM overhead
- Cons: No type-safe queries; manual migration management; higher risk of SQL injection; slower development velocity for 14-model schema
- Rejected: Team size (1 developer) favors productivity over control

### Prisma 7 vs Prisma 6
Prisma 7 uses driver adapters by default, replacing the built-in connection pool. This was chosen because:
- Driver adapter pattern is required for serverless environments (Vercel Edge, Cloudflare Workers)
- `@prisma/adapter-pg` with `pg` Pool provides explicit connection management
- v7 default `connectionTimeoutMillis: 0` (no timeout) differs from v6 default of 5s — acceptable for local dev; production should configure explicit timeouts

## Consequences
- Prisma generates TypeScript types and client from the schema — all queries are type-checked
- Prisma Migrate handles schema versioning (`prisma migrate dev` / `prisma migrate deploy`)
- Cannot use raw SQL for complex queries (Prisma limitation) — must use Prisma's query API
- Connection pooling configured via `pg` Pool options, not Prisma URL parameters
- In serverless (Vercel), a PgBouncer or connection pooler is required to prevent exhausting database connections
- Prisma 7's `adapter-pg` pattern is relatively new (2026) — documentation is less mature than v6
