# ADR-006: Stubbed LINE Notification Service

## Status
Accepted

## Date
2026-06-28

## Context
LINE Messaging API is the standard notification channel for Thai businesses — most customers use LINE daily. Requirements:
- Notify customers when a booking is created, status changes, or completed
- Prompt customers to leave a review after completed bookings/boat tours
- Must work without real LINE API credentials during development
- Must not block the booking flow if notification fails (fire-and-forget)

## Decision
Implement a `notification.service.ts` with two modes:
- **Mock mode** (default, when `LINE_CHANNEL_ACCESS_TOKEN` not set): Logs notifications to `console.log` — no API call made
- **Live mode** (when env vars are set): Sends messages via LINE Messaging API using `@line/bot-sdk`
- All calls from the booking service are fire-and-forget (no `await`, no error propagation)

```typescript
// Fire-and-forget pattern — booking flow never waits for notification
notifyBookingCreated(userId, 'room', { id: booking.id, totalPrice });
```

## Alternatives Considered

### Await notifications (blocking)
- Pros: Guaranteed delivery; errors are visible
- Cons: Booking creation, cancellation, and completion would be delayed by the LINE API round-trip (200-500ms); if LINE API is down, bookings cannot be created
- Rejected: Booking flow must succeed regardless of notification delivery

### Push notifications via WebSocket
- Pros: Real-time delivery, no external dependency
- Cons: Requires WebSocket server (not compatible with serverless); user must be on the website to receive
- Rejected: LINE is the primary communication channel for Thai customers

### Email notifications as primary
- Pros: Works without LINE account
- Cons: Email open rates in Thailand are lower than LINE; spam filtering may block transactional emails
- Rejected: LINE is more effective for the target market; email was already implemented for OTP (console.log stub)

### Logging service (Sentry, DataDog)
- Pros: Tracks notification delivery failures
- Cons: Adds third-party dependency and cost
- Rejected: Current scale doesn't warrant production monitoring

## Consequences
- Booking creation, cancellation, and completion hooks call the notification service
- In mock mode, notifications are invisible — developer must check server console logs
- Switching to live mode requires adding `LINE_CHANNEL_ACCESS_TOKEN` and `LINE_CHANNEL_SECRET` to environment variables
- Notification service is a separate module (`src/modules/notifications/`) with no routes — only a service file
- No notification retry logic — if the LINE API call fails, the notification is silently dropped
- Review prompts are sent for both room bookings (review RoomType) and boat bookings (review BoatType)
